# Detection Engineering – Custom Network Reconnaissance (Nmap) Rule

---

## 1. Overview

This phase focuses on designing and refining a custom detection rule to identify network reconnaissance activity, specifically port scanning behavior generated by tools such as Nmap.

Reconnaissance detection is critical as it represents the **initial stage of an attack lifecycle**, where attackers attempt to discover exposed services and vulnerabilities.

---

## 2. Objective

The objective of this phase is to:

* Detect port scanning behavior using custom logic
* Improve visibility into reconnaissance activity
* Reduce false positives through better tuning
* Strengthen early-stage attack detection

---

## 3. Detection Gap Identified

During earlier phases:

* Port scanning activity was detected using a custom rule (**100102**)
* However, detection relied primarily on repeated connection attempts
* Some legitimate or unrelated activities triggered similar alerts

### Identified Limitation

```text id="gap2"
Overlapping behavior patterns leading to potential false positives
```

---

## 4. Detection Strategy

To improve reconnaissance detection:

* A custom rule was designed to monitor repeated connection attempts across multiple ports
* Detection is based on **frequency and pattern of connections** from a single source IP
* The rule is tuned to trigger only when scanning behavior is clearly identified

---

## 5. Rule Logic

The custom rule is based on:

* Multiple connection attempts
* Activity across different ports
* High-frequency behavior within a short timeframe

---

## 6. Rule Configuration

### File Location (Wazuh Manager)

```bash id="loc2"
/var/ossec/etc/rules/local_rules.xml
```

---

### Custom Rule

```xml id="rule2"
<group name="custom_recon,network,">
  <rule id="100102" level="10" frequency="10" timeframe="60">
    <if_group>syslog</if_group>
    <description>Custom Port Scan Detection - Multiple connection attempts</description>
    <mitre>
      <id>T1046</id>
    </mitre>
  </rule>
</group>
```

---

## 7. Rule Explanation

* **Rule ID (100102)** → Custom reconnaissance detection rule
* **Level (10)** → Medium-high severity
* **Frequency (10)** → Triggers after 10 events
* **Timeframe (60)** → Within 60 seconds
* **if_group (syslog)** → Applies to system/network logs
* **MITRE ID (T1046)** → Network Service Discovery

---

## 8. Implementation Steps

### Step 1 – Open Rule File

## 9. Rule Testing

To validate the rule:

* An Nmap scan was executed from the attacker machine
* Multiple ports were probed on the target system
* System logs were monitored for connection attempts

---

## 10. Validation Results

The custom rule successfully triggered alerts indicating:

* High-frequency connection attempts
* Sequential probing of ports
* Reconnaissance activity from a single source IP

This confirms effective detection of scanning behavior.

---

## 11. Detection Enhancement Impact

This rule improves detection by:

* Identifying early-stage attacker behavior
* Providing visibility into reconnaissance activity
* Enabling proactive defense before exploitation

---

## 12. MITRE ATT&CK Mapping
